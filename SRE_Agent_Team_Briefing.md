Resumen ejecutivo  Qué vamos a construir y por qué  Vamos a construir un agente de SRE que automatiza el proceso completo de gestión de  incidentes en un e - commerce. Cuando alguien reporta un problema   —   ya sea un QA, un  monitor automático o soporte   —   hoy ese reporte llega desordenado, poco técnico y requiere  que un ingeniero lo analice manualmente, identifique el componente afectado, cree un ticket y  avise al equipo. Ese proceso puede tomar entre 10 y 40 minutos en producción. Nuestro agente  lo hace en menos de 60 segundos.  El diferenciador real frente a otros equipos no es solo usar un LLM: es que el agente entiende  la arquitectura interna del e - commerce. Usando RAG sobre el código fuente de Reaction  Commerce, puede mapear un reporte vago como "no puedo pagar" al plugin exac to afectado  (api - plugin - payments - stripe), al resolver GraphQL involucrado (placeOrder), y sugerir acciones  técnicas concretas. Eso convierte un reporte de usuario en una tarea de ingeniería accionable  automáticamente.  Frase clave para el demo  "Our system transforms unstructured incident reports into architecture - aware, actionable  engineering tasks by understanding the internal structure of a modern headless e - commerce  platform   —   in under 60 seconds."

02. El e - commerce que usaremos  Reaction Commerce (Mailchimp Open Commerce)  ¿Qué es?  Reaction Commerce (rebrandeado como Mailchimp Open Commerce en su v4) es una  plataforma de comercio headless, API - first y completamente modular. Toda la funcionalidad  está implementada como plugins independientes sobre un servidor GraphQL (Apollo Server).   El  repositorio es un monorepo gestionado con pnpm workspaces.  Fue discontinuado oficialmente en junio 2023 (v5.0.0 fue la última release), pero esto no nos  afecta en absoluto. Lo usamos como base de conocimiento para el agente: no lo corremos, lo  indexamos. Su arquitectura es madura, bien documentada y perfecta para   demostrar triage  architecture - aware.  Métricas del repo   Stack tecnológico  12,400 estrellas en GitHub   Node.js 14 – 16 + ES Modules  2,200 forks   GraphQL con Apollo Server  21,600 commits   MongoDB como base de datos  261 releases (última: v5.0.0)   Docker Compose incluido  Discontinuado: junio 2023   pnpm workspaces (monorepo)  Arquitectura interna   —   cómo funciona  Entender esto es esencial para que el agente sea inteligente. El flujo de una petición en  Reaction Commerce es:  1   Storefront / Admin hace una petición  El frontend llama al endpoint GraphQL en localhost:3000/graphql  2   Middleware de autenticación y autorización  Se valida el token y los permisos del usuario antes de continuar  3   Resolver del plugin correspondiente  Apollo Server enruta la operación al resolver registrado por el plugin responsable  4   Lógica del plugin (context.mutations / context.queries)  El plugin ejecuta la lógica de negocio usando el context compartido  5   MongoDB (context.collections)  Lectura o escritura en la base de datos. Los resolvers trabajan con IDs internos

03. Qué vamos a construir  Arquitectura completa del sistema  Flujo end - to - end  1   Usuario reporta el incidente en la UI  Formulario con: descripción, fuente (QA / soporte / monitoring), adjunto (imagen o log file)  2   Backend recibe y valida el reporte  FastAPI valida el input, aplica guardrails básicos anti - prompt injection y lanza el pipeline del  agente  3   Triage Agent analiza el incidente  LangGraph ejecuta el pipeline: clasifica, extrae entidades, consulta el RAG del codebase,  analiza adjuntos con visión, genera diagnóstico  4   Se crea el ticket automáticamente  GitHub Issues recibe el ticket con título, descripción técnica, severidad, componente  afectado y acciones sugeridas  5   Se notifica al equipo técnico  Slack webhook envía mensaje al canal del equipo correspondiente (payments - team, orders -  team, etc.)  6   Resolution watcher detecta el cierre  Polling o webhook sobre el ticket. Cuando pasa a 'resolved', se notifica al reportero original  por email  Pipeline del agente (el corazón del sistema)  El agente corre en LangGraph, donde cada nodo es una función con input/output tipado. El flujo  es secuencial con manejo de errores en cada paso.  Input:   { description,   source,   attachment }  ↓  Node   1:   Normalize   &   Validate   Input  →   sanitiza   texto,   valida   adjunto, detecta prompt injection  ↓  Node   2:   Classify   Incident  →   tipo:   checkout_failure   /   login_error / catalog_issue / ...  ↓  Node   3:   Extract   Entities  →   servicio   afectado,   feature,   error patterns mencionados  ↓  Node   4:   RAG   Query   sobre   Reaction Commerce

→   busca   en   packages/api - plugin - * el código relevante  →   retorna:   plugin   exacto,   resolver, archivo  ↓  Node   5:   Analyze   Attachment   (multimodal)  →   si   hay   imagen/log,   Gemini   extrae información técnica  ↓  Node   6:   Generate   Technical   Summary  →   resumen   técnico   accionable   con contexto del código  ↓  Node   7:   Assign   Severity  →   P1   (crítico)   /   P2   (alto)   /   P3 (medio)  ↓  Node   8:   Route   to   Team  →   payments - team   /   orders - team   / catalog - team / ...  ↓  Output:   JSON   estructurado   con   todo el diagnóstico  Output estructurado del agente  {  "incident_type":   "checkout_failure",  "severity":   "P1",  "affected_plugin":   "api - plugin - payments - stripe",  "layer":   "GraphQL   resolver   →   placeOrder",  "affected_file":   "resolvers/Mutation/placeOrder.js",  "assigned_team":   "payments - team",  "summary":   "Users   cannot   complete checkout due to Stripe payment timeout. División del trabajo  Quién hace qué   —   sin ambigüedad  La división está pensada para que cada persona tenga un dominio claro, pueda trabajar en  paralelo desde el día 1, y haya un punto de integración definido entre cada parte.  Persona 1   —   Frontend  • Formulario de reporte: descripción (textarea), fuente (QA / soporte / monitoring), adjunto (drag  & drop de imagen o .log)  • Vista de resultado: muestra el diagnóstico estructurado del agente (severidad, plugin afectado,  resumen, acciones)  • Vista de estado del ticket: link al ticket de GitHub Issues, estado actual  • Conexión con el backend: llamadas a POST /incident y GET /incident/:id  • Carpeta de trabajo: apps/frontend/  • Tech: Next.js 14 + Tailwind CSS + TypeScript  • Entrega Día 1: formulario funcional que envía al backend y muestra respuesta JSON  • Entrega Día 2: UI pulida, vista de estado, integración completa  Persona 2   —   Backend + API  • Endpoint POST /incident: recibe el formulario, valida inputs, lanza el pipeline del agente  • Endpoint GET /incident/:id: devuelve estado del incidente (en proceso / completado / error)  • Endpoint POST /webhook/ticket - update: recibe notificación cuando el ticket se resuelve  • Validación de inputs y guardrails anti - prompt injection (sanitizar texto antes de pasarlo al LLM)  • Modelo de datos del incidente (Pydantic) y manejo de errores del pipeline  • Carpeta de trabajo: apps/backend/  • Tech: FastAPI + Python + Pydantic  • Entrega Día 1: POST /incident retorna un JSON mock del agente (para que frontend pueda  conectar)  • Entrega Día 2: integración real con el agente, manejo de archivos adjuntos  Persona 3   —   Agente (IA)   —   el trabajo más crítico  • Pipeline completo en LangGraph: 8 nodos del classify al route

• Prompts para cada nodo (en archivos .txt separados para poder iterar rápido)  • Structured output en JSON usando Pydantic para garantizar el schema  • Análisis multimodal: integrar Gemini para procesar imágenes y logs adjuntos  • Integración con el módulo RAG (usa las funciones de rag/queries.py)  • Manejo de errores y retries en cada nodo  • Carpeta de trabajo: agent/  • Tech: LangGraph + Google Gemini API (o Claude vía OpenRouter)  • Entrega Día 1: pipeline funcional sin RAG   —   produce JSON estructurado dado texto libre  • Entrega Día 2: RAG integrado, multimodal activo, prompts refinados  Persona 4   —   RAG + Integraciones + Infra  • Indexar packages/api - plugin - * de Reaction Commerce en Chroma (rag/ingest_repo.py)  • Función de query RAG que dado un tipo de incidente devuelve el contexto de código relevante  • GitHub Issues API: crear ticket con metadata estructurada del agente  • Slack Webhook: enviar notificación formateada al canal correcto según el team asignado  • Resolution watcher: polling cada 30s sobre los tickets abiertos, notificar por email al resolver  • docker - compose.yml final que une todos los servicios (frontend, backend, vector - db)  • Langfuse setup: instrumentar el pipeline del agente para trazabilidad  • Los 4 archivos de documentación: README, AGENTS_USE.md, SCALING.md,  QUICKGUIDE.md  • Carpetas de trabajo: rag/ + integrations/ + observability/ + docker - compose.yml
